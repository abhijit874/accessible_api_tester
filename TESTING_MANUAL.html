Before You Start — Understanding the Basics

What is an API?

An API is a way for one computer program to talk to another. When you open a weather app and it shows today's temperature, it sent a request to a weather server and got data back. That request-and-response is an API call.

Think of it like a waiter in a restaurant: you (the app) tell the waiter (API) what you want, the waiter goes to the kitchen (server), and brings back your food (data).

What is JSON?

JSON is the format most APIs use to send data. It looks like this:

{
  "name": "Raj",
  "age": 28,
  "city": "Mumbai"
}

Curly braces {} hold the data. Each piece has a name (called a key) and a value, separated by a colon.

Query Parameters

What we are testing: Can the app add filters to a URL and return only matching results?

What are query parameters? Think of them like a search filter on a shopping website — when you filter by brand or price, those filters are added to the URL.

How they look in a URL:
Without filter: https://jsonplaceholder.typicode.com/posts — returns all 100 posts
With filter: https://jsonplaceholder.typicode.com/posts?userId=1 — returns only posts by user 1

The question mark ? separates the address from the filter. userId is the filter name and 1 is the filter value.
ℹ️
Instead of typing ?userId=1 directly into the URL, this app has a Query params group in the request form where you fill in the name and value separately. The app builds the full URL with the filter automatically when you activate Send — the URL text field itself stays unchanged.

Steps

  1. Find the Method dropdown and confirm it is set to GET
  2. Find the URL text field and type — without any question mark:

    https://jsonplaceholder.typicode.com/posts

  3. Navigate down the form to the group labelled "Query params" — your screen reader will announce "Query params group"
  4. Inside the Query params group, find the Query param 1 name text field and type: userId
  5. Tab to the Query param 1 value text field and type: 1
  6. Activate the Send button — the app automatically adds ?userId=1 to the URL before sending. The URL text field itself does not change; the filter is applied internally.

Expected Response

Status: 200 OK · In Section 2 you got 100 posts with no filter. Now you should get only 10 posts — all of them belonging to user 1. Every single item in the response will have "userId": 1.

Environment Variables

What we are testing: Can the app store a reusable value and use it inside requests?
Why use variables? Instead of typing the full base URL in every request, you save it once as a variable named base_url, then write {{base_url}} in your request. The app fills in the real value before sending.

Step A — Create the variable

  1. Activate Variables in the sidebar navigation
  2. Find the Variable name text field and type: base_url
  3. Tab to the Variable value text field and type: https://jsonplaceholder.typicode.com
  4. Activate the Save variable button

Step B — Use the variable in a request

  1. Activate Requests in the sidebar navigation
  2. Find the Method dropdown and confirm it is set to GET
  3. Find the URL text field and type the variable name in double curly braces instead of the full URL:

    {{base_url}}/posts/1

  4. Activate the Send button — the app replaces {{base_url}} with the real URL before sending

What we are testing: Can the app automatically check the response against rules you define?
What is an assertion? A rule you set before sending — for example "the status must be 200" or "the response must contain the word userId". After sending, the app checks these rules and tells you pass or fail automatically.

Step A — Set up passing assertions

  5. Inside the Assertions group, find the Expected status number field and type: 200

Step B — Trigger a failing assertion

  1. Navigate back to the "Assertions" group and find the Expected status number field — change the value to: 999

Mock Server

What we are testing: Can the app simulate a fake API endpoint that returns data you define?
What is a Mock Server? Sometimes the real API is not ready yet. A mock lets you create a fake endpoint that returns whatever you want — so you can test your work before the backend is built.

Step A — Create a mock route

  1. Activate Mock Server in the sidebar navigation — the Mock route editor form opens automatically
  2. Find the Mock name text field and type: Fake Users
  3. Find the Method dropdown and select GET
  4. Find the Path under /mock text field and type: /users
  5. Find the Status number field and type: 200
  6. Find the Response content type text field and type: application/json
  7. Find the Response body text area and type exactly this: 
    [
  { "id": 1, "name": "Alice" },
  { "id": 2, "name": "Bob" }
]
  8. Activate the Save mock button

Step B — Send a request to the mock

  1. Activate Requests in the sidebar navigation
  2. Find the Method dropdown and confirm it is set to GET
  3. Find the URL text field and type — this points to your local mock server, not the internet:

    http://127.0.0.1:5090/mock/users

  4. Activate the Send button
💡
The port number 5090 is the default for the desktop app. If it does not work, check the port number shown in the console when the app starts and use that instead.
